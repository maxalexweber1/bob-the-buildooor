# Buildooor Wallet — Implementation Plan

> A standalone Cardano self-custody **browser extension** (CIP-30), built on
> `@harmoniclabs/buildooor`. **Fully independent of ODATANO** — its own repo,
> its own chain providers. This document is the blueprint; implementation happens
> in this new repository. ODATANO serves only as a code reference for a few
> battle-tested building blocks (linked at the end).

---

## 1. Goal & Scope

A MetaMask-/Eternl-style browser extension for Cardano that:

- Creates/imports HD wallets from a BIP39 mnemonic (CIP-1852 derivation).
- Keeps keys **encrypted locally** and **signs client-side** (no server holds keys).
- Builds transactions with buildooor (ADA, native tokens, **Plutus spend/mint**, governance).
- Exposes itself to dApps as a **CIP-30** provider (`window.cardano.<name>`), including `signData` (CIP-8) and Conway governance (CIP-95).
- Fetches chain data directly from swappable providers (Blockfrost / Koios / Ogmios+Kupo).

**Full feature set including Plutus** is the target; delivery is phased (§11).

## 2. Tech Stack Decisions

| Area | Choice | Rationale |
|---|---|---|
| Tx building / crypto | **`@harmoniclabs/buildooor`** (re-exports ledger-ts, bip32_ed25519, crypto, cbor, plutus-data, plutus-machine) | Pure-TS, **no WASM** → enables strict MV3 CSP `script-src 'self'` (no `wasm-unsafe-eval`). Byte-preserving CBOR → correct `scriptDataHash` & COSE. |
| Mnemonic layer | **`@scure/bip39`** | The only real gap in the HarmonicLabs stack (no wordlist/`generateMnemonic`). Audited, pure-JS. |
| HW wallets (optional, later) | `@cardano-foundation/ledgerjs-hw-app-cardano` + `@ledgerhq/hw-transport-webhid`; Trezor via `@trezor/connect` | Keys never leave the device. |
| UI | **React + TypeScript** | Matches Nami/Lace/Yoroi; large component pool. |
| Build | **Vite + `@crxjs/vite-plugin`** | MV3-capable HMR, easy multi-entry (inpage/content/background/popup). |
| Manifest | **MV3** | Mandatory for Chrome Web Store. Firefox port via `browser.*` polyfill. |
| State (popup) | Zustand/Redux + `chrome.storage` as source of truth | SW is ephemeral (§5) — persistence belongs in `chrome.storage`. |

> **Bundling note:** the `@harmoniclabs/*` packages are CommonJS **without** an `exports`/`module` field.
> In Vite you may need `optimizeDeps.include: ['@harmoniclabs/buildooor', ...]`; production bundling
> works. Heavy `bigint` usage → modern targets (no ES5).

**Rejected:** CSL (`@emurgo/cardano-serialization-lib`) / CML — Rust→WASM, forces `'wasm-unsafe-eval'`
(larger attack surface + re-instantiation on every SW cold start). Lucid Evolution / MeshJS are
batteries-included but also sit on WASM and give less byte-level control.

## 4. MV3 Architecture & Message Bridge

**Four contexts** (all verified from research):

```
dApp page → inpage (MAIN world) → content (ISOLATED) → background SW → popup
window.cardano.x.enable()  postMessage   chrome.runtime    open popup    user approves
        Promise ◄─────────── postMessage ◄── runtime resp ◄── resolve ◄────────┘
```

Core rules:

1. **The provider must be injected into the page's MAIN world** (the dApp calls `window.cardano` from
   its own JS). Content scripts run isolated → cannot set the page's `window.cardano`.
   **Method A (recommended, cross-browser):** the content script injects a
   `<script src=chrome.runtime.getURL('inpage.js')>` tag, listed in `web_accessible_resources`,
   `run_at: "document_start"`.
2. **inpage ↔ content:** the only channel is `window.postMessage`. **Mandatory security filters:**
   `e.source === window`, a namespaced `target`, `id` correlation per request, and forward `e.origin`.
3. **content ↔ background:** `chrome.runtime` — for `enable`/signing use a **long-lived port**
   (`chrome.runtime.connect`), because the port keeps the SW alive during user approval (Chrome 114+).
4. **Approval UI:** `signTx`/`signData`/first `enable` open a popup window
   (`chrome.windows.create({type:'popup'})`); queue the pending request in `chrome.storage` (key = reqId),
   the popup reads it, the user decides, the answer flows back.

**MV3 manifest (core):**
```jsonc
{
  "manifest_version": 3,
  "background": { "service_worker": "background.js", "type": "module" },
  "action": { "default_popup": "popup.html" },
  "content_scripts": [{ "matches": ["<all_urls>"], "js": ["content.js"], "run_at": "document_start" }],
  "web_accessible_resources": [{ "resources": ["inpage.js"], "matches": ["<all_urls>"] }],
  "content_security_policy": { "extension_pages": "script-src 'self'; object-src 'self';" },
  "permissions": ["storage", "unlimitedStorage", "idle", "alarms"]
}
```
> CIP-30 injection needs **no** `host_permissions` (the content-script relay suffices). Chain-data
> providers, however, do: the public hosts (`*.koios.rest`, `*.blockfrost.io`) are declared, and a
> self-hosted host is requested at runtime via `optional_host_permissions`. Pure-JS stack → **no**
> `'wasm-unsafe-eval'` and the CSP also carries `frame-ancestors 'none'`.

## 5. Service Worker Lifecycle (critical)

The MV3 SW is terminated aggressively: after **30 s of inactivity**, on events > 5 min, on `fetch` > 30 s.
**Global variables are lost.** Consequences for the wallet core:

- **Never** keep decrypted keys / unlock state in SW module globals — they vanish after 30 s.
- Encrypted vault → `chrome.storage.local`. Decrypted keyring → only volatile in SW RAM.
- **Unlock across SW death:** cache the derived **encryption key (not the password)** in
  `chrome.storage.session` (stays in-memory, survives SW respawn, cleared on browser close;
  keep the default access level `TRUSTED_CONTEXTS`). Pattern: MetaMask `cacheEncryptionKey`/`submitEncryptionKey`.
- **Auto-lock timer** (`chrome.alarms` + `chrome.idle`) still mandatory, to bound exposure.
- Keep the port open during approval; no `fetch` calls > 30 s (keep provider timeouts shorter).

## 6. Key Management (Keyring)

**Stack:** `@scure/bip39` (mnemonic↔entropy) → buildooor `XPrv` (HD) → buildooor `Address`.

```ts
import { generateMnemonic, mnemonicToEntropy, validateMnemonic } from '@scure/bip39';
import { wordlist } from '@scure/bip39/wordlists/english';
import { XPrv, harden } from '@harmoniclabs/buildooor';
import { Address, Credential, StakeCredentials } from '@harmoniclabs/buildooor';
import { blake2b_224 } from '@harmoniclabs/buildooor';

// 1) Generate / import
const mnemonic = generateMnemonic(wordlist, 256);          // 24 words
const entropy  = mnemonicToEntropy(mnemonic, wordlist);    // Uint8Array

// 2) Root key (BIP32-Ed25519, Icarus)
const root = XPrv.fromEntropy(entropy);                    // optional passphrase

// 3) CIP-1852: m/1852'/1815'/account'/role/index
//    role 0 = external, 1 = change, 2 = staking, 3 = DRep (CIP-105)
function deriveKey(acct: number, role: number, idx: number): XPrv {
  return root.derive(harden(1852)).derive(harden(1815)).derive(harden(acct))
             .derive(role).derive(idx);
}
const payment = deriveKey(0, 0, 0);
const stake   = deriveKey(0, 2, 0);

// 4) Address (Base = payment+stake; network via mainnet()/testnet())
const payCred   = Credential.keyHash(blake2b_224(payment.public().toPubKeyBytes()));
const stakeCred = StakeCredentials.keyHash(blake2b_224(stake.public().toPubKeyBytes()));
const addr = Address.testnet(payCred, stakeCred).toString();   // addr_test1...
```

> **Shortcut** for single-address cases: `Address.fromEntropy(entropy, network, accountIndex, addressIndex)`
> or `Address.fromXPrv(...)` returns a base address directly (`payment …/0'/0/0`, stake `…/0'/2/0`).

**Account discovery / gap limit:** CIP-30 does not specify this — the wallet must do it itself
(de-facto **gap limit 20**, like BIP-44/CIP-1852). Derive external (role 0) and change (role 1) addresses
until 20 consecutive unused ones are found (check usage via the provider).

**Conway keys (CIP-105):** DRep `…/3/0`, committee-cold `…/4/0`, committee-hot `…/5/0`. Credential = `blake2b_224(pubkey)`.

## 7. Chain-Provider Abstraction (replaces the ODATANO backend)

buildooor provides coin selection (`keepRelevant`) and fees (`calcMinFee`, `getMinimumOutputLovelaces`),
but **no** chain data. Implement your own interface (mirrors buildooor `IProvider`/`ITxRunnerProvider`):

```ts
interface IChainProvider {
  getProtocolParameters(): Promise<ProtocolParameters>;
  getUtxos(address: string): Promise<UTxO[]>;
  resolveUtxos(refs: TxOutRef[]): Promise<UTxO[]>;   // for reference inputs / script utxos
  getGenesisInfos(): Promise<GenesisInfos>;          // slot↔POSIX (validity intervals)
  submitTx(txCbor: string): Promise<string>;         // → txHash
  evaluateTx?(txCbor: string): Promise<RedeemerExUnits[]>; // Plutus: Ogmios preferred
  awaitTxConfirmation?(txHash: string): Promise<boolean>;
}
```

Implementations: **Blockfrost** (REST, easiest start), **Koios** (free), **Ogmios+Kupo** (your own infra,
needed for authoritative `evaluateTransaction` ex-units on Plutus). Provider selectable in settings; the
network (preview/preprod/mainnet) determines endpoint + address prefix.

> **Plutus note:** for correct script ex-units you need an `evaluateTx` source (Ogmios). Without it you
> only get buildooor's local CEK estimate — riskier. Mirror ODATANO's 2-pass build (see References).

## 8. Transaction Building & Signing

**Build** — `TxBuilderRunner` (chainable) or `TxBuilder.build(args)`:

```ts
import { TxBuilder } from '@harmoniclabs/buildooor';
const builder = new TxBuilder(protocolParameters, genesisInfos);

const candidates = await provider.getUtxos(senderAddr);
const tx = builder.buildSync({
  inputs:  builder.keepRelevant(outputValue, candidates),   // coin selection
  outputs: [{ address: recipient, value: Value.lovelaces(amount) }],
  changeAddress: changeAddr,
  // Plutus: collaterals, scripts, datums, redeemers, referenceInputs, mints, requiredSigners
  // Tokens: Value.singleAsset(policy, name, qty)
  // Validity: validityIntervalStart / ttl via slot↔POSIX from genesisInfos
});
```

**Sign** — buildooor `Tx.signWith` (adds a VKeyWitness in place, only if the key is a required signer):

```ts
tx.signWith(payment);              // PrivateKey | Uint8Array | XPrv
// HW wallet: tx.addVKeyWitness(new VKeyWitness({ vkey, signature }))
const signedCbor = tx.toCbor().toString();
```

**`signData` (CIP-8 COSE_Sign1)** — HarmonicLabs provides the primitives, **not** COSE itself.
Build it yourself: assemble `Sig_structure = ["Signature1", protected_bstr, h'', payload]` via
`@harmoniclabs/cbor`, sign with `signExtendedEd25519_sync(sigStruct, payment.toPrivateKeyBytes())`,
return `COSE_Sign1` + `COSE_Key` as CBOR hex (`{signature, key}`). Headers: `alg=EdDSA(-8)`,
`"address"`=raw address bytes, unprotected `hashed:false`. **Template:** ODATANO `cose-verifier.ts`
(verification; for production use the same structures in reverse).

> **Caveat:** `signData` does **not** work on Ledger/Trezor (HW refuses arbitrary COSE payloads).
> For "Sign-in with Cardano" plan a tx-based login fallback.

## 9. CIP-30 Provider (dApp interface)

Injected object (MAIN world):
```ts
window.cardano.<name> = {
  apiVersion: "1", name, icon,            // icon = data-URI
  supportedExtensions: [{ cip: 95 }],
  isEnabled(): Promise<boolean>,
  enable(opts?): Promise<API>,            // consent gate → popup
};
```
After `enable()` (origin persisted in the allowlist) the full API is granted. **All returns are hex-CBOR; accept bech32 OR hex for inputs.**

| Method | Source | Note |
|---|---|---|
| `getExtensions()` | Keyring/settings | Returns the `Extension[]` actually enabled for this connection (subset of `supportedExtensions`). Part of the full API — do not omit. |
| `getNetworkId()` | Keyring/settings | 0=testnet, 1=mainnet |
| `getUtxos(amount?, paginate?)` | Provider | `amount` set → only enough UTxOs to reach the target value; otherwise all; `null` if unattainable |
| `getBalance()` | Provider | `cbor<value>` |
| `getUsedAddresses` / `getUnusedAddresses` / `getChangeAddress` / `getRewardAddresses` | Keyring | hex address bytes |
| `getCollateral({amount})` | Provider | **DEPRECATED** in CIP-30 in favour of **CIP-40** collateral-output tx fields, but still implement (ADA-only UTxOs, ~5 ADA) — dApps in the wild continue to call it. |
| `signTx(tx, partialSign?)` | Signer | **Return only the `transaction_witness_set`**, not the whole tx. Per-call consent! |
| `signData(addr, payload)` | COSE (§8) | `{signature, key}` |
| `submitTx(tx)` | Provider | → `hash32` |

**Implement error codes exactly:** `APIError{InvalidRequest:-1, InternalError:-2, Refused:-3, AccountChange:-4}`,
`TxSignError{ProofGeneration:1, UserDeclined:2}`, `DataSignError{ProofGeneration:1, AddressNotPK:2, UserDeclined:3}`,
`TxSendError{Refused:1, Failure:2}`, `Paginate{page,limit}` / `PaginateError{maxSize}`.

**CIP-95 (Conway/governance):** via `enable({extensions:[{cip:95}]})`. Methods under `api.cip95.*`:
`getPubDRepKey()`, `getRegisteredPubStakeKeys()`, `getUnregisteredPubStakeKeys()`, `signData(addr|DRepID, payload)`;
`signTx` must witness Conway certs (vote delegation, DRep reg/retire) and voting/proposal procedures.
> `getRegisteredPubStakeKeys()` is intentionally **un-namespaced** (no `cip95.` prefix) per the CIP-95 spec — keep that exact name.

**Extension negotiation is generic.** An extension is a plain `{ cip: N }` object (integer, **no leading zeros**),
queried at runtime via `getExtensions()`, and its endpoints live under `api.cipNN.*` (e.g. `api.cip95.*`). Build the
inpage provider's extension dispatch **generically** off `supportedExtensions`, not hard-wired to CIP-95 — every future
capability rides this same mechanism. The official CIP-30 extensions register currently lists **exactly four**:

| Ext | Purpose | Status | Scope for us |
|---|---|---|---|
| **CIP-95** | Conway governance | Active | **Must-have** — GovTool refuses to connect without it; shipped by Eternl, Lace, Yoroi, NuFi, Vespr, Typhon. |
| **CIP-103** | Bulk / chained tx signing (`api.cip103.signTxs` / `submitTxs`) | Active | **Should-have** — only if we target DeFi/batch dApps; one approval for a chain of txs. See §14 defer decision. |
| **CIP-104** | Account public key | Proposed | Nice-to-have. |
| **CIP-106** | Multisig (native-script) — **disables** `signTx`/`signData`, replaces with `submitUnsignedTx`/`getCompletedTx` | Proposed | Out of scope while we are single-sig. Pairs with CIP-1854. |

## 10. Security (binding requirements)

1. **Vault encryption:** prefer **Argon2id** (OWASP min `m=19456 KiB, t=2, p=1`, higher for a seed).
   If Web-Crypto/FIPS only: PBKDF2-HMAC-SHA256 **≥600,000** iterations. Store KDF params in vault metadata
   (forward migration). Cipher **AES-256-GCM**, 32-byte random salt, fresh ≥12-byte IV per encryption.
   > **Divergence from the legacy Cardano precedent — recorded deliberately.** The historical wallet vault
   > standard is **EMIP-003** (Daedalus/Yoroi): **PBKDF2-HMAC-SHA512 at 19,162 iterations** + **ChaCha20Poly1305**
   > AEAD. We diverge on both axes on purpose: (a) **AES-256-GCM** over ChaCha20Poly1305 — both are sound AEADs;
   > AES-GCM is available directly in `crypto.subtle` (no extra dependency, honours CLAUDE.md §1.9), whereas a
   > pure-JS ChaCha20Poly1305 would add attack surface. (b) **Iteration count:** EMIP-003's 19,162 is *far* below
   > current OWASP guidance — we target **≥600,000** PBKDF2-HMAC-SHA256 (or Argon2id), an order of magnitude higher.
   > Net: our vault is **not** EMIP-003 wire-compatible (we cannot import a raw Daedalus/Yoroi secret-key blob without
   > a dedicated converter), which is acceptable because import is via **mnemonic** (CIP-1852), not raw vault blob.
2. **Storage:** encrypted vault in `chrome.storage.local` (or IndexedDB). **Never `localStorage`**
   (unavailable in the SW, synchronous, trivially readable). Decrypted keyring only in SW RAM. Key cache in `storage.session`.
3. **Per-call consent:** CIP-30 mandates it — confirm **every** `signTx`/`signData` via the trusted popup.
   A content script must **never** obtain a signature without user approval. Sign only in the privileged context.
4. **Decode the transaction human-readably** (inputs/outputs/amounts/mint/metadata/certs) **before** approval —
   consent alone does not prevent blind-signing drains. (WalletProbe 2025: 39/39 wallets abusable.)
5. **Origin allowlist** in the background store; pass `e.origin` through the bridge and enforce it.
6. **Popup hygiene:** untrusted strings (token names, dApp titles, metadata) only via `textContent`,
   length-limited; show the full destination address + paste warning (clipboard hijackers).
7. **Supply chain:** lockfiles, pinned deps, **scoped/audited npm publish tokens**, dependency sandboxing
   (LavaMoat-style). Ledger Connect-Kit incident: one compromised dependency drains even HW-wallet users.
8. **Strict CSP** (`script-src 'self'`) retained — the strongest structural defense against injected popup code.

- Full `enable()` flow + approval popup + origin allowlist.
- All CIP-30 methods incl. `signTx` (witness set!), `submitTx`, error codes, pagination.

**M5 — Plutus (full scope)**
- Spend/mint: datums/redeemers (`plutus-data`), collateral mgmt, reference inputs (CIP-31)/inline datums (CIP-32)/reference scripts (CIP-33).

**M6 — Governance (CIP-95) & HW wallets**
- CIP-95 extension (`api.cip95.*`), Conway certs/voting. DRep keys (CIP-105).
- Ledger (WebHID) / Trezor (Connect, popup mode; keep the transport **outside** the SW — page/offscreen).

**M7 — Hardening & store release**
- Security review, LavaMoat, pen-test popup/content; Firefox port (`browser.*` polyfill); CWS listing.

## 12. CIP Build Order (condensed)

`CIP-1852 + CIP-105` (HD core) → `CIP-30 + CIP-8` (bridge/signing) → `CIP-95` (governance extension)
→ `CIP-31/32/33` (tx decoder for dApp txs) + `CIP-25/68` (NFT display). `CIP-1694` as the governance domain model.

**Optional / later (status from the cardano-foundation/CIPs register, verified 2026-06):**
- `CIP-103` (bulk tx signing) — *Active*, **should-have** for DeFi UX; defer decision in §14.
- `CIP-1854` (multisig HD) + `CIP-106` (multisig connector) — *Active / Proposed*, **only if multisig becomes a product goal**; both out of scope for single-sig v1.
- `CIP-104` (account pub-key extension) — *Proposed*, nice-to-have.
- `CIP-113` (programmable tokens) — *Proposed; assessed 2026-07-17*, **not compatible today, deliberately deferred**.
  Regulated assets (stablecoins/securities) whose transfer logic runs on-chain. Architecturally inverted vs.
  normal assets: ALL holders share one script payment credential (the "programmable logic base"); ownership is
  the **stake-credential slot** of `addr(programmable_logic_base, owner_credential)` — so our payment-key-based
  discovery never sees them, and transfers need a script spend + two withdraw-zero invocations + registry
  reference inputs + a **stake-key** witness (not payment-key). Most of the required Plutus machinery already
  exists from M5 (`plutusBuild.ts`); the work is discovery + registry client + wiring + approval decode.
  **Full task breakdown: EXECUTION_PLAN M9 (T9.1–T9.5).** Gated on upstream maturity — see §14.

**Asset / metadata / messaging CIPs (standards review round 2, 2026-06 — all *Active*, primary-source confirmed).**
These are *display/build* concerns, not connector methods — they ride the tx decoder, asset views, and
Plutus build. Priority for a self-custody wallet:

| CIP | Label / fields | Priority | Where it lands |
|---|---|---|---|
| **CIP-40** Collateral output | tx-body `collateral_return` (16) + `total_collateral` (17) | **MUST** (Plutus/script spend) | M5 — **covered**: `plutusBuild.ts` passes `collaterals` + `changeAddress` to buildooor's `TxBuilder.buildSync`, which sets both fields. Supersedes the CIP-30 `getCollateral` pattern. |
| **CIP-25** NFT metadata (v1/v2) | metadata label `721`, mint-time | **SHOULD ✅ DONE** | `provider.getAssetMetadata` (Blockfrost `/assets/{asset}` + Koios `/asset_info`) → name/description; dashboard shows the resolved name (lazy, persisted 24 h cache, network-keyed). **Images ✅** (A2): SW proxies the art → `data:` URI under a tightened `img-src 'self' data:` CSP, with SSRF-validated fetch (`core/assetImage.ts`). |
| **CIP-68** Datum metadata | ref-NFT `100` + user `222`/`333`/`444`, same policy | **SHOULD ✅ NAMES DONE** | Same `getAssetMetadata` path (Blockfrost decodes CIP-68 → `onchain_metadata`). Name prefix also handled locally (CIP-67 ✅). Koios/Ogmios metadata fetch not implemented (Blockfrost-only for now). |
| **CIP-20** Tx message/memo | metadata label `674`, `msg:[…]` ≤64 B/line | **SHOULD ✅ DONE** | `core/tx/txMessage.ts` decodes label 674 (+ CIP-83 `enc` flag); shown in the signTx approval (`MessageRows`). Outbound "attach memo" not yet built. |
| **CIP-67** Asset-name label registry | 4-byte name prefix `[0000\|label\|CRC8\|0000]` | NICE ✅ DONE | `core/cip67.ts` (`parseCip67` with CRC-8/SMBUS validation, `cip67LabelName`); `balance.ts` strips the prefix for the display name + exposes `cip67Label`; badge in dashboard + approval. |
| **CIP-83** Encrypted tx messages | addendum to CIP-20, label `674` + `enc` (basic = AES-256-CBC) | NICE | optional decode of encrypted memos; non-decoding just shows ciphertext. |
| **CIP-36 / CIP-15** Catalyst voting registration | metadata `61284`/`61285` | NICE (open) | only if we add Catalyst voting — unresolved in the review; treat as out-of-scope until product asks. |

> CIP-40 vs `getCollateral`: we keep the deprecated `getCollateral` handler for dApp compatibility
> (T4.8) **and** build correct collateral-output fields ourselves on Plutus spends (M5).

## 14. Open Decisions / Risks

- **Argon2id in the browser:** SubtleCrypto has no Argon2 → a WASM build (`argon2-browser`/`hash-wasm`)
  would force `'wasm-unsafe-eval'` and thereby negate the CSP advantage of the pure-JS stack.
  **Trade-off:** either PBKDF2-≥600k (Web Crypto, no CSP compromise) **or** Argon2id accepts a narrow
  `'wasm-unsafe-eval'`. Decide early. **Resolved in EXECUTION_PLAN T1.1 → PBKDF2-≥600k.** The legacy
  Cardano precedent (EMIP-003: PBKDF2-SHA512 @ 19,162 + ChaCha20Poly1305) is **not** adopted — see §10.1
  for the recorded divergence and rationale.
  > **Standards review confirmation (2026-06, OWASP Password Storage Cheat Sheet):** OWASP ranks
  > Argon2id first (`m=19456 KiB, t=2, p=1`), scrypt second, and lists **PBKDF2-HMAC-SHA256 @ 600,000
  > as the FIPS-140-aligned choice**. Our PBKDF2-SHA256-≥600k **exactly meets OWASP's PBKDF2 floor**.
  > Argon2id is the only "better" tier, and a pure-JS Argon2id either pulls in WASM (violates §1.9) or
  > is too slow — so **PBKDF2-≥600k stays the default**. Re-open only if an audited pure-JS Argon2id
  > compatible with `script-src 'self'` appears. (Open: the round did not extract the *current* shipped
  > KDF numbers from Lace/Eternl/Nami/Yoroi — OWASP is the anchor, not a competitor's parameter.)
- **CIP-103 (bulk tx signing) — defer decision.** *Active* extension, but real-world dApp adoption of
  `api.cip103.signTxs()` is unverified (open question from the standards review). **Decision: defer to
  post-v1**, behind the generic extension-dispatch mechanism (§9) so it is a pure add-on when a target dApp
  actually requires it — no architectural debt from waiting. Re-evaluate once a concrete DeFi integration asks for it.
- **CIP-113 programmable tokens — read-only first, transfers gated (2026-07-17).** The Cardano Foundation
  reference implementation ([cip113-programmable-tokens](https://github.com/cardano-foundation/cip113-programmable-tokens))
  is explicitly R&D-grade: unaudited, contracts may still change — which would change the
  `programmable_logic_base` script hash and registry policy that all wallet-side logic keys on.
  **Decisions:** (a) build discovery/display (EXECUTION_PLAN T9.1–T9.3) only against preview/preprod, NO
  mainnet constants until an audited deployment exists; (b) transfer building (T9.4–T9.5) additionally needs
  a recorded human sign-off (T1.1-style), since it introduces stake-key witnessing of script spends — a new
  signing path (§10.3/§10.4 apply in full); (c) **CIP-30 exposure:** programmable-token UTxOs are excluded
  from dApp-facing `getBalance`/`getUtxos`/`getCollateral` (they are not vkey-spendable; including them would
  poison dApp coin selection) — popup display only, revisit if a CIP standardizes exposure; (d) registry-node
  UTxO refs are resolved fresh at build time, never cached (upstream-documented pitfall). Re-verify the spec
  status (CIPs PR #444) and per-network constants at implementation time.
- **Plutus ex-units without your own Ogmios infra:** Blockfrost/Koios don't offer a full `evaluateTransaction`
  like Ogmios → plan Ogmios+Kupo for M5, or ensure a provider eval endpoint.
- **buildooor CJS bundling in an MV3 SW (`type:module`):** verify interop early in M0.
- **Firefox parity:** WebHID/WebUSB are missing → HW wallets there only via Trezor Bridge; test MAIN-world script-tag injection.
- **`signData` HW incompatibility:** the login flow needs a tx-based fallback.
